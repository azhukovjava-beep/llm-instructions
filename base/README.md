# Base

`base/` - это канонический слой, в котором в нормализованном виде отражается все важное из поддерживаемых провайдеров.

Идея простая:

- не хранить тут финальные платформенные файлы как источник правды;
- хранить тут общий смысл, общие сущности и правила переноса;
- отдельно фиксировать, как каждая платформа выражает этот смысл в своем формате.

## Что должно отражаться в base

В `base/` должна попадать по сути вся полезная модель из провайдеров:

- типы инструкций;
- агентские роли и подагенты;
- skills, plugins, rules, custom instructions и похожие сущности;
- ограничения форматов;
- области действия файлов;
- иерархия и наследование инструкций;
- особенности размещения файлов в репозитории;
- все различия, которые влияют на синхронизацию.

При этом все это нужно хранить не как набор разрозненных provider-specific файлов, а как общую модель с понятным переводом в каждую платформу.

## Принцип организации

`base/` делится на три слоя:

1. Нормализованная модель.
2. Описание провайдеров и их особенностей.
3. Правила преобразования из общей модели в платформенные файлы.

## Структура

```text
base/
|-- README.md
|-- schema/            # общая нормализованная модель сущностей
|-- providers/         # провайдер-специфичные возможности и ограничения
|-- guidelines/        # общие правила разработки, ревью, безопасности, стиля
|-- agents/            # общие агентские роли и профили
|-- instructions/      # общие инструкции и политики
|-- templates/         # шаблоны сборки платформенных файлов
|-- mappings/          # соответствие schema -> platform outputs
|-- sync/              # правила синхронизации, сценарии, чеклисты
`-- docs-sources/      # ссылки на официальную документацию и заметки по ней
```

## Назначение папок

- `schema/` - единый словарь сущностей, из которого потом строятся платформенные файлы.
- `providers/` - как именно Claude, Codex, Cursor и другие провайдеры реализуют похожие сущности.
- `guidelines/` - общие инженерные правила.
- `agents/` - нормализованное описание ролей, агентов и подагентов.
- `instructions/` - общие operational instructions, policies и conventions.
- `templates/` - шаблоны генерации выходных файлов платформ.
- `mappings/` - правила, по которым общая сущность уходит в конкретные файлы конкретной платформы.
- `sync/` - логика обновления, сравнения и разруливания конфликтов.
- `docs-sources/` - канонические ссылки на официальную документацию, по которой проверяется актуальность.

## Что хранить как нормализованные сущности

Хорошая база должна описывать не просто markdown-тексты, а сущности:

- instruction;
- rule;
- agent;
- skill;
- memory source;
- template;
- provider capability;
- provider limitation;
- sync policy;
- mapping rule;
- platform exception.

## Что не стоит хранить в base

- финальные платформенные артефакты как единственный источник правды;
- дубли одного и того же текста без нормализации;
- случайные provider-specific файлы без объяснения, как они соотносятся с общей моделью;
- устаревшие выдержки из документации без ссылки на официальный источник.

## Правило обновления

Если изменение можно обобщить, его надо сначала оформить в `base/`.

Если изменение пока пришло только из одной платформы:

1. нужно определить, можно ли его поднять в общую модель;
2. если можно - сначала обновляется `base/`;
3. потом обновляются остальные платформы;
4. если нельзя - создается платформенное исключение с объяснением причины.
