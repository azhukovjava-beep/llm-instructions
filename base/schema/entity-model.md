# Entity Model

Этот файл задает каноническую модель сущностей для `base/`.

## Канонические типы сущностей

- `instruction-set` - основной набор инструкций для проекта, команды или области работы.
- `rule` - атомарное правило или ограничение.
- `rule-set` - набор правил, который может жить в отдельном артефакте.
- `agent-profile` - специализированный исполнитель с собственной ролью, ограничениями и trigger-описанием.
- `skill` - переиспользуемый workflow или playbook с инструкциями и дополнительными файлами.
- `memory-source` - источник постоянного контекста или накопленных знаний.
- `approval-policy` - правила разрешений, sandbox и command-approval.
- `hook` - детерминированная автоматизация вокруг событий жизненного цикла.
- `template` - шаблон выходного платформенного файла.
- `provider-capability` - возможность, которую поддерживает конкретная платформа.
- `provider-limitation` - ограничение платформы.
- `platform-exception` - осознанное отклонение от общей модели для конкретной платформы.

## Обязательные поля сущности

Любая нормализованная сущность в `base/` по возможности должна иметь:

- `id` - стабильный идентификатор.
- `kind` - один из канонических типов.
- `title` - короткое человекочитаемое имя.
- `summary` - краткое описание смысла.
- `body` - полный нормализованный смысл.
- `scope` - где применяется сущность.
- `load-mode` - как именно сущность должна попадать в контекст.
- `targets` - в какие платформы она переносится.
- `provider-bindings` - как она отображается в конкретных провайдерах.
- `source-links` - ссылки на официальную документацию или первоисточник.
- `exceptions` - список известных platform-specific отклонений.
- `last-reviewed` - дата последней ручной или автоматической проверки.

## Канонический scope

Используем единый словарь scope:

- `enterprise`
- `user`
- `repository`
- `directory`
- `path-pattern`
- `task`
- `session`
- `machine-local`

## Канонический load-mode

Используем единый словарь режимов загрузки:

- `always`
- `startup-chain`
- `on-demand`
- `path-triggered`
- `manual-invocation`
- `implicit-invocation`
- `event-driven`
- `policy-evaluated`

## Как думать о нормализации

Примеры перевода из провайдеров в общую модель:

- `CLAUDE.md` -> `instruction-set`
- `AGENTS.md` -> `instruction-set`
- `.cursor/rules/*.mdc` -> `rule-set`
- `.claude/skills/<name>/SKILL.md` -> `skill`
- `.agents/skills/<name>/SKILL.md` -> `skill`
- `.claude/agents/*.md` -> `agent-profile`
- `~/.claude/projects/<project>/memory/` -> `memory-source`
- `codex/rules/*.rules` -> `approval-policy`

## Практическое правило

Если новая провайдерская сущность не укладывается в текущую модель, сначала надо расширить модель здесь, а не городить исключение сразу в платформенной папке.
