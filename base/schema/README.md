# Schema

`schema/` описывает общую модель, в которую приводятся сущности всех провайдеров.

Это самый важный слой `base/`: именно здесь мы переводим разные форматы в единый словарь.

## Зачем это нужно

У разных платформ похожие идеи называются по-разному:

- `AGENTS.md`
- `CLAUDE.md`
- `.cursor/rules`
- provider-specific skills
- provider-specific agents
- memory files
- repository instructions

Если хранить только финальные файлы платформ, синхронизация быстро расползется. Поэтому здесь мы описываем общий смысл в одной модели.

## Минимальные сущности

- `instruction` - общий текст поведения или правила.
- `rule` - более узкое ограничение или обязательное требование.
- `agent` - роль, профиль или специализированный исполнитель.
- `skill` - подключаемое расширение поведения или workflow.
- `template` - заготовка для генерации провайдерского файла.
- `mapping` - правило переноса общей сущности в платформу.
- `exception` - причина, по которой перенос в одну из платформ отличается.
- `capability` - что платформа умеет.
- `limitation` - чего платформа не умеет или требует по-особому.

## Нормализация

Каждая сущность в идеале должна иметь:

- стабильный идентификатор;
- тип;
- краткое описание;
- полный нормализованный текст;
- scope;
- приоритет;
- теги;
- список целевых платформ;
- список исключений;
- ссылки на источники в `../docs-sources/`;
- правила маппинга из `../mappings/`.

## Scope

Общая модель должна уметь выражать scope:

- global;
- repository;
- directory;
- file-type;
- task-specific;
- provider-specific exception.

## Практическое правило

Если что-то важно больше чем для одной платформы, оно сначала должно получить представление в `schema/`, а уже потом уходить в `claude/`, `codex/` или `cursor/`.
