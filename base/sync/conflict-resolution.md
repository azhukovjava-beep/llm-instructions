# Conflict Resolution

Этот файл нужен для случаев, когда платформы расходятся.

## Типы конфликтов

- `format-conflict` - у платформ разный допустимый формат.
- `scope-conflict` - одна платформа умеет path-scoped instructions, другая нет.
- `capability-conflict` - у одной платформы есть native feature, у другой нет.
- `behavior-conflict` - одинаковый артефакт читается по-разному.
- `policy-conflict` - разные approval / sandbox semantics.

## Правило решения

1. Сначала сохраняем общий смысл в `base/schema/`.
2. Потом описываем ограничения в `base/providers/`.
3. Затем фиксируем перенос в `base/mappings/`.
4. Если перенос без потерь невозможен, создаем `platform-exception`.

## Что писать в platform-exception

- какая сущность не переносится полностью;
- в какой платформе проблема;
- почему;
- какой выбран fallback;
- нужно ли вернуться к этому решению при следующей проверке документации.
